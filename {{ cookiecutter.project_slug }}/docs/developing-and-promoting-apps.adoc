= Developing & Promoting Apps

This guide covers the day-to-day workflow on the {{ cookiecutter.platform_name }}
platform: developing an application on dev, adding a brand-new application, and
promoting changes from dev -> stage -> prod. Read
link:how-it-works.adoc[How It Works] first if you have not.

== The mental model: two things you promote

A deployment is the combination of a *chart version* and a *values file*. You
promote each one independently:

[cols="1,3",options="header"]
|===
| What | Lives in | Promote by

| Chart version
| `bootstrap/appset-apps.yaml` (`targetRevision` per element)
| Releasing a tag in `{{ cookiecutter.repo_charts }}`, then moving the tag
  forward for stage, then for prod.

| Configuration (values)
| `clusters/<cluster>/apps/<appName>.yaml`
| Editing the next environment's values file to match what you proved out in the
  one below it.
|===

The environments form a ladder:

[cols="1,1,2",options="header"]
|===
| Environment | Chart revision | Intent

| `{{ cookiecutter.cluster_dev }}`
| `main` (branch)
| Always-latest. Chart changes appear here automatically.

| `{{ cookiecutter.cluster_stage }}`
| pinned tag (ahead of prod)
| Soak a release before it reaches prod.

| `{{ cookiecutter.cluster_prod }}`
| pinned tag
| Only proven releases.
|===

== Develop on dev

`{{ cookiecutter.cluster_dev }}` tracks the charts repo's `main` branch, so the
fast loop is:

. Make chart changes in `{{ cookiecutter.repo_charts }}` and merge them to
  `main`.
. Argo CD syncs the new chart to dev automatically (no change needed in this
  repo).
. Tune dev behaviour by editing
  `clusters/{{ cookiecutter.cluster_dev }}/apps/<appName>.yaml` and committing.

To change *only* configuration (replicas, log level, resources, UI, ...) without
a new chart, just edit the dev values file - the chart revision stays `main`.

== Add a brand-new application

Say you want to add an app called `myapp`.

. *Create the chart* in `{{ cookiecutter.repo_charts }}` at `charts/myapp/`, and
  cut release tags for the environments that pin one (for example
  `myapp-1.0.0`).
. *Add one element per cluster* to the `list` generator in
  `bootstrap/appset-apps.yaml`:
+
[source,yaml]
----
- appName: myapp
  cluster: {{ cookiecutter.cluster_dev }}
  environment: dev
  releaseName: myapp-dev
  targetRevision: main
- appName: myapp
  cluster: {{ cookiecutter.cluster_stage }}
  environment: stage
  releaseName: myapp-stage
  targetRevision: myapp-1.0.0
- appName: myapp
  cluster: {{ cookiecutter.cluster_prod }}
  environment: prod
  releaseName: myapp-prod
  targetRevision: myapp-1.0.0
----
. *Add a values file for every cluster* (required - `ignoreMissingValueFiles` is
  `false` for apps):
+
[source,text]
----
clusters/{{ cookiecutter.cluster_dev }}/apps/myapp.yaml
clusters/{{ cookiecutter.cluster_stage }}/apps/myapp.yaml
clusters/{{ cookiecutter.cluster_prod }}/apps/myapp.yaml
----
. *Commit and push* to `{{ cookiecutter.repo_appsets_branch }}`. The
  `applications` `ApplicationSet` generates `myapp-dev`, `myapp-stage`, and
  `myapp-prod`, and Argo CD syncs them.

== Promote a chart version (dev -> stage -> prod)

This is the core release flow. dev already runs `main`, so promotion is about
moving stage and prod onto a released tag.

. *Cut a release tag* in `{{ cookiecutter.repo_charts }}` from a known-good
  commit (the one validated on dev), for example `podinfo-6.15.0`.
. *Promote to stage* - in `bootstrap/appset-apps.yaml`, bump the stage element's
  `targetRevision` to the new tag:
+
[source,diff]
----
  - appName: podinfo
    cluster: {{ cookiecutter.cluster_stage }}
    environment: stage
    releaseName: podinfo-stage
-   targetRevision: podinfo-6.14.0
+   targetRevision: podinfo-6.15.0
----
Commit, push, and let it soak. Verify `podinfo-stage` is healthy in Argo CD.
. *Promote to prod* - once stage looks good, bump the prod element's
  `targetRevision` to the *same* tag:
+
[source,diff]
----
  - appName: podinfo
    cluster: {{ cookiecutter.cluster_prod }}
    environment: prod
    releaseName: podinfo-prod
-   targetRevision: podinfo-6.13.0
+   targetRevision: podinfo-6.15.0
----
Commit and push. Argo CD rolls prod onto the proven release.

Keep stage at least one release ahead of prod so there is always a soak step.

== Promote configuration changes

Values changes ride the same ladder, just in the per-cluster files instead of
the appset. After proving a setting on dev:

. Copy the relevant change from
  `clusters/{{ cookiecutter.cluster_dev }}/apps/<appName>.yaml` into
  `clusters/{{ cookiecutter.cluster_stage }}/apps/<appName>.yaml`, adjusting for
  stage's scale.
. After it soaks, apply the equivalent change to
  `clusters/{{ cookiecutter.cluster_prod }}/apps/<appName>.yaml`.

Do not blindly copy whole files between environments - replica counts, resource
limits, and HPA settings are intentionally larger as you move up the ladder.
Promote the *change*, not the entire file.

== Rollback

Because every environment's state is a Git commit, rollback is a revert:

* *Chart version* - set the affected element's `targetRevision` back to the
  previous tag in `bootstrap/appset-apps.yaml` and push. (For dev, revert the
  offending commit on the charts repo's `main`.)
* *Configuration* - `git revert` the values commit, or edit the values file back
  to the known-good state and push.

Self-heal and auto-sync mean the cluster converges to whatever Git says, so the
recovery action is always "make Git correct again".

== Quick reference

[cols="1,2",options="header"]
|===
| Task | Action

| Change dev to latest chart
| Merge to `main` in `{{ cookiecutter.repo_charts }}` (automatic).

| Change an app's config on one cluster
| Edit `clusters/<cluster>/apps/<appName>.yaml`.

| Promote a release to stage
| Bump the stage element's `targetRevision` in `bootstrap/appset-apps.yaml`.

| Promote a release to prod
| Bump the prod element's `targetRevision` to the same tag.

| Add a new app
| Add 3 elements to `bootstrap/appset-apps.yaml` + 3 values files under
  `clusters/`, and create the chart in `{{ cookiecutter.repo_charts }}`.

| Roll back
| Revert the `targetRevision` or values commit in Git.
|===
